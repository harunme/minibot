## 2026-03-27 — Kids-Chat Skill 推迟：从 V1.0 移至 V2.0

### 背景

原路线图将 Kids-Chat Skill（儿童友好对话规则、安全内容过滤）放在 V1.0 M4 中。经讨论认为：Kids-Chat 属于体验优化类需求，不影响核心对话链路的技术验证，MVP 阶段不做优化类需求。

### 决策变更

| 项目 | 旧方案 | 新方案 |
|------|--------|--------|
| Kids-Chat Skill | V1.0 M4（与全链路联调一起交付） | **V2.0**（知识库与内容管理阶段） |

### 核心理由

1. **MVP 只验证核心链路**：V1.0 的目标是跑通"语音输入 → STT → Agent → TTS → 语音输出"全链路，Kids-Chat 不影响这个验证
2. **优化类往后排**：儿童友好对话规则和安全内容过滤属于体验优化，遵循"最小 MVP → 抓核心问题 → 优化类往后排"原则
3. **与 V2 内容管理天然相关**：Kids-Chat 的对话规则和内容过滤与知识库/内容管理场景更贴合，放在 V2 更合理

### 影响

- V1.0 M4 从"全链路联调 + Kids-Chat"精简为"全链路联调"
- V1.0 范围表移除 Kids-Chat Skill 行
- V2.0 范围表新增 Kids-Chat Skill（P1）

## 2026-03-27 — 多租户+管理后台推迟：从 V1.0b 移至 V3.5

### 背景

原路线图将 V1.0 拆为 V1.0a（核心对话链路）和 V1.0b（多租户+管理后台），计划在核心对话链路跑通后紧接着做多租户和管理后台。经讨论认为：多租户和管理后台属于运营基础设施，应在软件和硬件都完成 MVP 验证后再实施，避免过早优化。

### 决策变更

| 项目 | 旧方案 | 新方案 |
|------|--------|--------|
| V1.0 范围 | V1.0a（对话链路）+ V1.0b（多租户+管理后台） | **仅核心对话链路**（单租户 MVP） |
| 多租户+管理后台 | V1.0b（紧跟 V1.0a） | **V3.5**（独立版本，在 V1 + V3 之后） |

### 核心理由

1. **先验证核心再补运营**：多租户和管理后台不影响核心技术可行性验证，单租户即可跑通全链路
2. **软硬件 MVP 优先**：应先确认软件对话链路（V1）和硬件原型（V3）都跑通，再投入多租户架构
3. **避免过早优化**：如果核心链路或硬件验证不通过，多租户的投入就浪费了
4. **V3.5 可结合真实硬件验证**：推迟到硬件原型就绪后，多设备路由可用真实设备验证，更有价值

### 影响

- V1.0 里程碑从 M1-M7 精简为 M1-M4
- 新增 V3.5 版本（多租户+管理后台），依赖 V1 + V3
- V5a 依赖从 V1.0 管理后台改为 V3.5
- FastAPI、React、aiosqlite 等技术栈移至 V3.5 引入

## 2026-03-27 — 硬件方案选型：ESP32-S3 + 外挂 4G/GPS 模组

### 背景

DECISIONS.md §七 待确认事项中列有「硬件方案选型（ESP32 vs Linux 方案），推迟到 V3 详细设计时确定」。经过 ESP32 全系列评估，现提前给出选型建议，为 V1 测试客户端模拟和 V3 硬件适配打下基础。

### 决策结论

| 项目 | 选型 | 备注 |
|------|------|------|
| **主控 SoC** | **ESP32-S3-WROOM-1-N16R8** | 双核 Xtensa LX7 @240MHz, 16MB Flash, 8MB PSRAM |
| 4G 通信模组 | **SIMCom A7670C**（或合宙 Air724UG） | LTE Cat.1, UART 接 ESP32-S3, 低成本稳定 |
| GPS 定位模块 | **中科微 AT6558** 或 u-blox NEO-6M | 集成在 4G 模组中或独立 UART |
| 音频输入 | **INMP441**（I2S 数字 MEMS 麦克风） | I2S 直连 ESP32-S3, 内置降噪, 16kHz/16bit |
| 音频输出 | **MAX98357A**（I2S 数字功放） | I2S 直连 ESP32-S3, 3W 输出, 驱动小扬声器 |
| 音频编解码 | **Opus** 编解码（软件实现） | ESP32-S3 双核跑 Opus 没压力, 与 V1 WebSocket Channel 音频格式一致 |
| 电池管理 | TP4056 + DW01 | 锂电池充放电管理, 支持 USB-C 充电 |
| 蓝牙配网 | ESP32-S3 内置 BLE 5.0 | 初始配网 + SIM/WiFi 配置 |

### ESP32 系列对比及选型理由

#### 候选芯片对比

| 特性 | ESP32 (经典) | ESP32-S3 | ESP32-C3 | ESP32-C6 |
|------|-------------|----------|----------|----------|
| CPU 核心 | 双核 Xtensa LX6 @240MHz | **双核 Xtensa LX7 @240MHz** | 单核 RISC-V @160MHz | 单核 RISC-V @160MHz |
| SRAM | 520KB | **512KB** | 400KB | 512KB |
| PSRAM 支持 | 最大 4MB | **最大 8MB** (Octal SPI) | 不支持 | 不支持 |
| I2S 接口 | 2 个 | **2 个** | 1 个 | 1 个 |
| Wi-Fi | 4 (2.4GHz) | **4 (2.4GHz)** | 4 (2.4GHz) | **6 (2.4GHz)** |
| 蓝牙 | Classic + BLE 4.2 | **BLE 5.0** | BLE 5.0 | BLE 5.0 + 802.15.4 |
| USB | 无 | **USB-OTG + JTAG** | 无 | USB Serial |
| AI 加速 | 无 | **有 (向量指令)** | 无 | 无 |
| GPIO 数量 | 34 | **45** | 22 | 30 |
| 功耗 (Active) | ~240mA | ~240mA | ~130mA | ~140mA |
| Deep Sleep | ~10μA | ~7μA | ~5μA | ~7μA |
| 价格 (模组) | ¥12-18 | **¥18-28** | ¥8-12 | ¥14-20 |
| 成熟度 | 非常成熟 | **成熟** | 成熟 | 较新 |

#### 选择 ESP32-S3 的核心理由

1. **双核 + Opus 编解码**：Opus 编解码器需要较高算力。ESP32-S3 双核 LX7 @240MHz 可以一核跑音频编解码 + WebSocket 通信，另一核跑业务逻辑和 4G 模组控制，不互相阻塞
2. **8MB PSRAM**：音频缓冲区是内存大户。16kHz/16bit PCM 每秒 32KB，需要缓冲 Opus 编码前的原始 PCM、解码后的播放 PCM、WebSocket 发送/接收缓冲。512KB SRAM 不够用，8MB PSRAM 给足裕量
3. **2 个 I2S 接口**：一路 I2S 接 INMP441 麦克风（输入），另一路 I2S 接 MAX98357A 功放（输出），同时采集和播放无需分时复用
4. **BLE 5.0 配网**：家长手机 APP/小程序通过 BLE 向设备推送 WiFi 密码、4G APN 配置、服务器地址等
5. **USB-OTG**：开发阶段可通过 USB 直接烧录和调试，量产时支持 USB-C 充电
6. **ESP-IDF 网络支持**：`esp-websocket-client` 组件成熟稳定，对接 V1 WebSocket 服务端
7. **AI 向量指令**：预留未来端侧 VAD（语音活动检测）、关键词唤醒等轻量 AI 能力

#### 为什么不选其他型号

| 不选 | 原因 |
|------|------|
| ESP32 (经典) | 无 PSRAM Octal SPI、无 USB-OTG、BLE 仅 4.2、GPIO 偏少，扩展性受限 |
| ESP32-C3 | 单核 + 无 PSRAM + 仅 1 路 I2S，无法满足音频双向同时工作 + Opus 编解码的算力需求 |
| ESP32-C6 | 单核 + 无 PSRAM，Wi-Fi 6 和 802.15.4 对本项目无价值，算力不足 |
| Linux 方案 (全志/RK) | 成本高（¥50+）、功耗大（500mA+）、启动慢（10s+）、对电池不友好，杀鸡用牛刀 |

### 具体型号选择：ESP32-S3-WROOM-1-N16R8

| 参数 | 值 |
|------|------|
| Flash | 16MB (Quad SPI) |
| PSRAM | 8MB (Octal SPI) |
| 天线 | PCB 板载天线 |
| 温度范围 | -40°C ~ 85°C |
| 封装 | 18mm × 25.5mm × 3.1mm |
| 参考价 | ~¥22 (单片) |

选择 N16R8 理由：
- **16MB Flash**：OTA 双分区（A/B）需要约 2×4MB，剩余空间存储音频缓存、配置、证书等
- **8MB PSRAM**：音频处理缓冲 + 网络收发缓冲 + Opus codec 工作内存，8MB 足够宽裕
- PCB 天线版本体积更小、成本更低，适合产品内嵌

### 外围模组方案

#### 4G 通信：SIMCom A7670C

| 参数 | 说明 |
|------|------|
| 制式 | LTE Cat.1（上行 5Mbps / 下行 10Mbps） |
| 接口 | UART (AT 指令) + USB |
| 供电 | 3.4V~4.2V, 峰值电流 ~2A |
| 尺寸 | 24mm × 24mm × 2.3mm |
| GPS | 可选内置 GPS（A7670C-GNSS 版本） |
| 价格 | ~¥25-35 |

**为什么选 Cat.1 而不是 Cat.4**：
- MiniBot 音频上传带宽需求约 16-32kbps（Opus 编码后），Cat.1 的 5Mbps 上行绰绰有余
- Cat.1 模组功耗更低（待机 ~1mA vs Cat.4 ~3mA）
- Cat.1 价格更低（便宜 ¥20-30）
- 全国 4G Cat.1 网络覆盖已非常成熟

**备选方案**：合宙 Air724UG（Cat.1, 更低价），或 SIMCom SIM7600CE（Cat.4, 如需更高带宽预留）。

#### GPS 定位

- **优先方案**：选择 A7670C-GNSS 版本，4G + GPS 一体，减少 BOM 和 PCB 面积
- **备选方案**：独立 GPS 模块（AT6558 / NEO-6M），UART 接入 ESP32-S3

#### 音频 I/O

```
                    I2S Bus 0                      I2S Bus 1
  INMP441 (Mic) ──────────────→ ESP32-S3 ───────────────→ MAX98357A (Amp) → Speaker
  (16kHz/16bit)    BCK,WS,SD                BCK,WS,SD       (3W Class-D)
```

| 组件 | 型号 | 接口 | 价格 | 说明 |
|------|------|------|------|------|
| 麦克风 | INMP441 | I2S | ~¥3 | 数字 MEMS, -26 dBFS SNR 61dB, 内置抗混叠滤波 |
| 功放 | MAX98357A | I2S | ~¥4 | 3W Class-D, 无需外部 DAC, 内置 PLL |
| 扬声器 | 1.5W~3W 腔体喇叭 | 导线 | ~¥3-5 | 40mm 直径，适合儿童产品 |

#### 电池与充电

| 组件 | 方案 | 说明 |
|------|------|------|
| 电池 | 3.7V 锂聚合物, 2000~3000mAh | 续航目标 4-6 小时连续使用 |
| 充电 IC | TP4056 | 1A 线性充电, USB-C 输入 |
| 电池保护 | DW01 + 8205A | 过充/过放/过流/短路保护 |
| 升压/稳压 | ME6211 (3.3V LDO) | ESP32-S3 供电 |
| 4G 模组供电 | SY8088 (4.0V DCDC) | A7670C 需要 3.4~4.2V, 峰值 2A |

#### 续航估算

| 状态 | 电流 (估算) | 说明 |
|------|------------|------|
| 活跃对话 | ~350mA | ESP32-S3 + 4G 传输 + 麦克风 + 扬声器 |
| 待机监听 | ~80mA | ESP32-S3 Light Sleep + 4G 待机 + VAD 唤醒 |
| 深度休眠 | ~2mA | ESP32-S3 Deep Sleep + 4G PSM |

以 2500mAh 电池为例：
- 纯对话（假设 50% 活跃 + 50% 待机）：约 **11-12 小时**
- 日常使用（假设 20% 活跃 + 80% 待机）：约 **20+ 小时**

## 2026-03-27 — 语音方案统一：火山引擎 ASR/TTS

### 背景

V1 初始设计中 TTS 主选方案在多处文档间存在不一致（CosyVoice2 / MiniMax / 火山引擎），ASR 缺少独立决策记录。经讨论确认统一技术栈。

### 决策变更

| 项目 | 旧方案 | 新方案 |
|------|--------|--------|
| TTS (V1) | CosyVoice2 主选 + MiniMax 备选 | **火山引擎 TTS**（WebSocket 流式） |
| ASR (V1) | 火山引擎 ASR（无独立决策记录） | **火山引擎 ASR**（WebSocket 流式，独立决策） |

### 核心理由

1. **统一技术栈**：ASR + TTS 全部使用火山引擎，减少对接成本和运维复杂度
2. **WebSocket 流式一致性**：ASR/TTS 都走 WebSocket 流式协议，架构统一
3. **中文效果优先**：火山引擎在中文语音场景有丰富音色和优化

### 扩展策略

- `ASRProvider` / `TTSProvider` **抽象基类保留多提供商扩展能力**
- ASR 未来可扩展阿里等厂商（只需新增 Provider 实现 + 配置切换）
- TTS 未来可扩展阿里等厂商（只需新增 Provider 实现 + 配置切换）
- V4 音色克隆使用云厂商 API（火山引擎定制化音色 / 阿里 CosyVoice API 等），通过 TTSProvider 抽象层接入，不自部署模型

## 2026-03-27 — 架构优化：双协议通信方案

> ⚠️ **已废弃**：此决策在 2026-03-30 被取代。V1 已统一使用 WebSocket 通道，参考 xiaozhi-esp32-server 架构。
> MQTT Channel 代码已删除，详见 `websocket-channel.md`。以下内容保留作为决策历史。

### 背景

在 V1 初始设计中，硬件与服务端之间选择了 WebSocket 作为通信协议。经过进一步讨论，确认 IoT 场景的最佳实践是 **MQTT + WebSocket 双协议互补**，而非单一 WebSocket。

### 决策变更

| 项目 | 旧方案 | 新方案 |
|------|--------|--------|
| ESP32 ↔ 后端 | WebSocket | **MQTT** |
| 后端 ↔ 火山引擎 ASR/TTS | (未明确) | **WebSocket** |

### 架构分工

```
ESP32 硬件设备 ──[MQTT]──→ MiniBot 后端 ──[WebSocket]──→ 火山引擎 ASR/TTS
  │                           │
  │ • 设备状态上报             │ • 音频流实时识别（ASR）
  │ • 音频分片传输             │ • 音频流实时合成（TTS）
  │ • 控制指令下发             │ • 流式双向通信
  │ • QoS 消息保证             │
  │ • 低功耗、弱网稳定          │
```

### 选择 MQTT 负责设备接入的理由

1. **低功耗**：协议头仅 2 字节起，适合电池供电的 ESP32 设备，显著延长续航
2. **弱网稳定**：QoS 0/1/2 三级保证，SIM 卡 4G 信号不稳定时也能可靠传输
3. **断线重连**：原生 persistent session 支持，自动恢复订阅和消息队列
4. **Topic 路由**：天然适合多设备管理（如 `device/{device_id}/audio/up`），与多租户架构契合
5. **遗嘱消息（LWT）**：设备异常断线时自动通知后端，便于状态监控
6. **ESP32 生态成熟**：ESP-IDF 原生支持 MQTT，有大量参考实现
7. **音频分片传输**：MQTT 支持二进制 payload，配合 QoS1 保证音频帧不丢失

### 选择 WebSocket 负责后端对接 AI 服务的理由

1. **流式交互**：火山引擎 ASR/TTS 均提供 WebSocket 流式 API，天然匹配
2. **实时性**：全双工通信，边发送音频边接收识别/合成结果
3. **nanobot 已有基础**：框架内多个 Channel 已使用 `websockets` 库
4. **协议标准**：HTTP 升级握手，穿透性好，与后端微服务架构兼容

### 对 V1 设计的影响

- 硬件 Channel 从 `WebSocket Channel` 改为 `MQTT Channel`
- 新增 MQTT Broker 部署（推荐 EMQX 或 Mosquitto）
- 新增后端 → 火山引擎的 WebSocket 客户端（ASR/TTS 流式调用）
- Topic 设计和 MQTT 帧协议需重新定义
- STT 方案调整为火山引擎 ASR（统一技术栈）

### 待确认事项

- [ ] 火山引擎 ASR/TTS WebSocket API 的具体接入方式和定价
- [x] ~~MQTT Broker 选型~~ — V1 开发用 Mosquitto，生产切 EMQX；**统一使用 JWT Token 认证**，确保迁移只需换 Broker 配置
- [x] ~~MQTT QoS 级别选择~~ — 音频数据 QoS 0（低延迟），控制/状态消息 QoS 1（可靠），详见 mqtt-channel.md §3.3

## 2026-03-27 — 项目启动讨论

### 一、项目背景

基于现有开源项目 **nanobot**（超轻量级个人 AI 助手框架，v0.1.4.post5）进行扩展，开发一款面向**儿童**的智能语音聊天硬件产品的软件后端系统。

产品形态：**软硬件结合**的儿童语音伴侣。

### 二、核心需求

#### 2.1 用户场景

```
小朋友对着硬件说话
  → 硬件采集语音，通过网络上传
  → 服务端：STT 语音转文字 → Agent 处理（结合知识库）→ TTS 文字转语音
  → 服务端将语音回传硬件
  → 硬件音箱播放回复
```

#### 2.2 功能要求

| 编号 | 功能 | 说明 |
|------|------|------|
| F1 | 语音对话 | 小朋友说话 → AI 语音回复，全程语音交互 |
| F2 | 知识库回复 | 回复内容基于家长上传的知识库（故事、绘本内容等） |
| F3 | 音频播放 | 根据小朋友要求播放家长内置的音乐或故事绘本音频 |
| F4 | 音色克隆 | 回复音色模拟家长声音，增强亲切感 |
| F5 | 多租户 | 支持多个家庭使用，数据隔离 |
| F6 | 管理后台 | 家长上传 MP3、PDF 等文件，管理内容和设备 |

#### 2.3 硬件要求

| 组件 | 要求 |
|------|------|
| 麦克风 | 语音采集，支持降噪 |
| 音箱 | 语音播放，音质清晰 |
| 电池 | 续航要强，方便携带 |
| 蓝牙 | 用于配网和连接 |
| GPS 定位 | 儿童定位安全功能 |
| SIM 卡 | 独立联网，不依赖 WiFi |

### 三、技术决策

#### 3.1 硬件通信协议

**决策：WebSocket 统一通信协议**（⚠️ 已更新，见 2026-03-30 架构调整）

- **ESP32 ↔ 后端**：WebSocket — 参考 xiaozhi-esp32-server 架构，直连无需中间件
- **后端 ↔ 火山引擎 ASR/TTS**：WebSocket — 高实时性、流式传输，负责音频流实时识别与合成
- 原始方案（MQTT + WebSocket 双协议）已废弃，理由：统一 WebSocket 简化架构，ESP32 原生支持 WebSocket

#### 3.2 TTS 语音合成

**决策：火山引擎 TTS，统一技术栈**

| 阶段 | 方案 | 说明 |
|------|------|------|
| V1 | 火山引擎 TTS（WebSocket 流式） | 与 ASR 统一技术栈，中文音色丰富，流式合成低延迟 |
| V4 | 云厂商音色克隆 API（火山引擎/阿里等） | 对接音色克隆 API，复用 TTSProvider 抽象层，无需自部署 GPU |

**设计要点**：
- 定义 `TTSProvider` 抽象基类，接口 `synthesize(text, voice_id) -> AsyncIterator[bytes]`
- 抽象层支持多提供商扩展，V1 实现火山引擎，未来可扩展阿里等厂商
- 支持流式输出，降低首字节延迟
- V1 火山引擎 TTS 快速验证，V4 扩展音色克隆 API Provider 实现家长音色

#### 3.3 ASR 语音识别

**决策：火山引擎 ASR（WebSocket 流式），统一技术栈**

| 阶段 | 方案 | 说明 |
|------|------|------|
| V1 | 火山引擎 ASR（WebSocket 流式） | 中文优化，流式识别，与 TTS 统一技术栈 |

**设计要点**：
- 定义 `ASRProvider` 抽象基类，接口 `recognize(audio) -> str` + `recognize_stream(audio_stream) -> AsyncIterator[str]`
- 抽象层支持多提供商扩展，V1 实现火山引擎，未来可扩展阿里等厂商

#### 3.4 知识库/RAG

**决策：ChromaDB 向量数据库 + Embedding**

- ChromaDB：轻量嵌入式，Python 原生，适配 SQLite 多租户模式
- Embedding：text2vec-base-chinese 或 DashScope Embedding API
- 每租户独立 ChromaDB collection，数据隔离
- 通过 nanobot Tool 机制注册 `knowledge_search` 工具

#### 3.5 多租户数据存储

**决策：SQLite 多数据库（每租户一个文件）**

- 理由：轻量、零依赖、天然隔离、易备份迁移
- 每个家庭拥有独立的：
  - SQLite 数据库文件（用户、设备、内容元数据）
  - ChromaDB collection（向量检索空间）
  - 文件存储目录（音频、PDF）
  - nanobot workspace（会话、记忆、技能）
- 租户路由：device_id → tenant_id 映射

### 四、nanobot 现有能力评估

| 维度 | 评级 | 说明 |
|------|------|------|
| 架构清晰度 | ★★★★★ | 消息总线解耦，模块职责分明 |
| LLM 集成 | ★★★★★ | 20+ Provider，覆盖中外主流 |
| Channel 支持 | ★★★★★ | 12 渠道 + 插件系统 |
| STT | ★★★☆☆ | V1 新增火山引擎 ASR |
| TTS | ☆☆☆☆☆ | 未集成，V1 新增火山引擎 TTS |
| 知识库/RAG | ★★☆☆☆ | 仅文件+记忆整合，无向量检索 |
| 多租户 | ★★☆☆☆ | 仅多实例隔离，无共享多租户 |

### 五、架构策略

**核心原则：扩展不改核心**

- 通过 Channel 插件机制接入硬件设备
- 通过 Provider 抽象层扩展 TTS 能力
- 通过 Tool 注册机制扩展知识库检索
- 新增代码以独立模块组织，不修改 nanobot 核心文件（loop.py、runner.py 等）
- 配置扩展遵循现有 camelCase 约定
